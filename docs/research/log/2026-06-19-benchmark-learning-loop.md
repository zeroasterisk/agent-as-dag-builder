# 2026-06-19 -- Prototype 10: Benchmark + Learning Loop

## Summary

Built a benchmark harness with 23 test cases and an LLM-as-judge scorer, then ran 3 iterations of a learning loop that analyzes results, proposes DAG improvements, and generates updated configs. The learning loop improved scores from v1 to v2 but regressed on v3, proving that automated optimization needs validation gates to prevent over-fitting.

## Benchmark design

**23 test cases** across three categories:
- Billing: 8 cases (double charges, refunds, bill increases, payment updates, cancellation, unauthorized charges, billing dates, discounts)
- Technical: 8 cases (connectivity, app crashes, login issues, slow speeds, 404 errors, streaming, notifications, 2FA)
- General: 7 cases (business hours, phone contact, referral programs, feedback, business services, mobile app, plan upgrades)

Each case has an expected category and quality criteria describing what a good response should contain.

## Scoring methodology

Each response is scored on 100 points across three dimensions:

| Dimension | Points | Method |
|---|---|---|
| Category accuracy | 40 | Deterministic: does the classifier route to the correct handler? |
| Quality | 40 | LLM-as-judge: does the response meet the quality criteria? |
| Helpfulness | 20 | LLM-as-judge: is the response concise, clear, and actionable? |

The judge model (`gemini-2.5-flash`) evaluates quality and helpfulness independently from the DAG's own execution model, avoiding self-evaluation bias.

## Learning loop results

Three iterations were run. After each iteration, the system:
1. Analyzes results (category breakdown, weakest cases)
2. Proposes 2-4 specific improvements using an LLM
3. Generates an updated YAML config with improvements applied
4. Writes the new config to disk for the next iteration

**Score progression:**

| Iteration | Aggregate Score | Category Accuracy | Quality | Helpfulness |
|---|---|---|---|---|
| v1 (baseline) | 92.3 | 100% | 35.2/40 | 17.1/20 |
| v2 (improved) | 93.8 (+1.5) | 100% | 36.1/40 | 17.7/20 |
| v3 (regressed) | 90.5 (-3.3) | 100% | 33.8/40 | 16.7/20 |

## Analysis

**v1 to v2 (+1.5 points):** The learning loop correctly identified weak cases and improved handler instructions. Quality and helpfulness both increased. Category accuracy remained at 100%.

**v2 to v3 (-3.3 points):** The third iteration over-optimized for the weakest cases from v2, making instructions too specific and reducing generality. The quality score dropped by 2.3 points across the board. This is a classic over-fitting pattern.

## Key insight: promotion gates are necessary

The learning loop can improve DAG performance, but without validation gates it can also degrade it. The v3 regression proves the need for:

1. **Canary testing**: Deploy improved configs to a subset of traffic before full promotion
2. **Regression detection**: Compare new version scores against the previous version, not just against absolute thresholds
3. **Rollback triggers**: Automatically revert if aggregate score drops below the previous version
4. **Validation holdout**: Reserve some test cases for validation only (not used in the learning analysis) to detect over-fitting

This matches the canary deployment pattern described in the design spec's BDD scenarios (Group 4: DAG Mutation).

## Artifacts produced

- `sandbox/10_benchmark_learning.py` -- Full benchmark + learning loop implementation
- `sandbox/customer_support_adk_v2.yaml` -- Improved config (generated by learning loop)
- `sandbox/customer_support_adk_v3.yaml` -- Over-fit config (generated by learning loop, regressed)

## Key takeaways

1. LLM-as-judge scoring provides meaningful signal for evaluating DAG quality
2. The learning loop successfully identifies weak cases and generates targeted improvements
3. Automated optimization without validation gates leads to over-fitting after 2-3 iterations
4. The promotion gate pattern from the design spec is validated as necessary, not optional
5. A holdout validation set is needed to detect over-fitting before promotion
