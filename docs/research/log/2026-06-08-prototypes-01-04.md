# 2026-06-08 -- Prototypes 01-04: ADK Workflow from Config

## Summary

Built four sandbox prototypes to validate the YAML-to-ADK-Workflow path. Proved that ADK 2.0 Workflow can be constructed programmatically from config dicts, but discovered that ADK's conditional routing has limitations that require workarounds.

## Prototype 01: ADK Workflow from config dict

**File:** `sandbox/01_adk_workflow_from_config.py`

Built an ADK `Workflow` from a Python config dict with nodes and edges. Each node became an `LlmAgent`, and edges were assembled as `(source, target)` tuples including the `START` sentinel.

**Result:** Workflow built and executed successfully. A simple classify-then-solve math pipeline ran end-to-end. This proved the YAML-to-ADK-Workflow path is feasible.

## Prototype 02: Conditional edges with feature flags (INCOMPLETE)

**File:** `sandbox/02_conditional_edges_INCOMPLETE.md`

Attempted to use ADK Workflow's routing map syntax `(node, {'value': target_node})` for conditional edge dispatch. The syntax was accepted by the framework but routing did not work correctly -- the downstream agent received no conversation context and produced no output.

**What worked:**
- `FunctionNode` with `parameter_binding='state'` reads session state
- Routing map syntax compiles without errors

**What did not work:**
- The routed-to agent produced no text output
- The function return value may need specific typing or edge handling that was not documented

## Prototype 03: YAML config loader

**File:** `sandbox/03_yaml_config_loader.py`

Built the core proof: load a YAML config file, build an ADK Workflow, and run it. Implemented `load_dag_from_yaml()` and `build_workflow()` functions. Conditional edges were parsed but not yet dispatched (deferred to prototype 04).

**Result:** Successfully loaded `customer_support.yaml`, built a workflow with classify + three handler agents, and ran test queries through it. The unconditional path (classify -> first handler) worked; conditional routing was deferred.

## Prototype 04: Conditional routing workaround

**File:** `sandbox/04_conditional_routing_workaround.py`

Since ADK 2.0's routing map syntax did not dispatch correctly (prototype 02), we tested a workaround: use `LlmAgent` with `sub_agents` for conditional routing. The parent agent classifies the request and delegates to the appropriate sub-agent using ADK's built-in `transfer_to_agent` tool.

**Result:** The sub-agent routing pattern worked. The router agent successfully classified queries and transferred to billing, technical, or general handlers. This is a viable workaround until ADK fixes or documents the routing map edge syntax.

## Key takeaways

1. ADK Workflow can be constructed programmatically from config -- the YAML-to-ADK path is proven
2. ADK's conditional routing via routing maps appears incomplete or undocumented in the current release
3. Sub-agent delegation via `transfer_to_agent` is a working alternative for conditional routing
4. The YAML config format defined in the design spec maps cleanly to ADK primitives
